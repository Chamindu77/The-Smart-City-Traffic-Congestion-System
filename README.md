## 🎯 Project Overview

A real-world end-to-end **Lambda Architecture** data pipeline that simulates IoT traffic sensors at **4 major Colombo junctions**, processes sensor data in real time to detect congestion, and generates nightly reports recommending where traffic police should be deployed.

### Problem Statement
Colombo's traffic management relies on manual observation, which cannot scale to handle the volume of data generated by modern urban traffic across multiple junctions simultaneously.

### Solution
A fully automated pipeline that:
- **Ingests** sensor data from 4 junctions every second via Apache Kafka
- **Detects** dangerous congestion immediately when `avg_speed < 10 km/h`
- **Calculates** Congestion Index using 5-minute tumbling windows
- **Generates** nightly police deployment reports via Apache Airflow
- **Stores** all data in PostgreSQL for historical analysis

### Junctions Monitored
| Junction | Location | Normal Speed Range |
|---|---|---|
| Junction_Pettah | Pettah, Colombo | 15 – 55 km/h |
| Junction_Kollupitiya | Kollupitiya, Colombo | 20 – 60 km/h |
| Junction_Nugegoda | Nugegoda | 10 – 50 km/h |
| Junction_Maharagama | Maharagama | 15 – 45 km/h |

## 🔧 Pipeline Components

### 1. Kafka Producer (`producer/producer.py`)

Simulates 4 IoT traffic sensors. Each junction publishes one JSON message per second to the `traffic-raw` Kafka topic.

```json
{
  "sensor_id": "Junction_Pettah",
  "timestamp": "2026-05-09T19:41:12.985773",
  "vehicle_count": 47,
  "avg_speed": 34.2
}
```

**Controlled alert injection:**
- Alert chance: `0.5%` per message
- Hard cap: maximum `3 critical events per junction per run`
- This satisfies the assignment requirement: *"critical traffic mimicking data generated occasionally"*

### 2. Spark Structured Streaming (`spark_jobs/stream_processor.py`)

Three parallel streaming queries:

| Query | Trigger | Sink | Purpose |
|---|---|---|---|
| Q1 Raw Events | every 10s | `traffic_events` | Historical store |
| Q2 Alerts | every 5s | `critical_traffic` | Speed < 10 km/h alert |
| Q3 Windows | every 30s | `congestion_index` | 5-min tumbling window |

**Congestion Index formula:**
```
congestion_index = total_vehicles / avg_speed
```
Higher index = heavier congestion.

### 3. Airflow DAG (`airflow/dags/traffic_dag.py`)

Scheduled nightly at `00:00` (`cron: 0 0 * * *`).

```
aggregate_peak_hour → generate_report → save_to_database
```

**To trigger manually:**
```bash
docker exec -it airflow-scheduler airflow dags trigger smart_city_traffic_nightly_report
```

## 🗄️ Database Schema

```sql
-- Raw sensor readings (written by Spark Q1)
CREATE TABLE traffic_events (
    id              SERIAL PRIMARY KEY,
    sensor_id       VARCHAR(50),
    event_timestamp TIMESTAMP,
    vehicle_count   INTEGER,
    avg_speed       FLOAT,
    ingested_at     TIMESTAMP DEFAULT NOW()
);

-- Critical alerts: speed < 10 km/h (written by Spark Q2)
CREATE TABLE critical_traffic (
    id              SERIAL PRIMARY KEY,
    sensor_id       VARCHAR(50),
    event_timestamp TIMESTAMP,
    vehicle_count   INTEGER,
    avg_speed       FLOAT,
    alert_message   TEXT,
    alerted_at      TIMESTAMP DEFAULT NOW()
);

-- 5-minute window aggregations (written by Spark Q3)
CREATE TABLE congestion_index (
    id              SERIAL PRIMARY KEY,
    sensor_id       VARCHAR(50),
    window_start    TIMESTAMP,
    window_end      TIMESTAMP,
    total_vehicles  INTEGER,
    avg_speed       FLOAT,
    congestion_idx  FLOAT,
    created_at      TIMESTAMP DEFAULT NOW()
);

-- Airflow nightly batch output
CREATE TABLE daily_peak_report (
    id              SERIAL PRIMARY KEY,
    report_date     DATE,
    sensor_id       VARCHAR(50),
    peak_hour       INTEGER,
    peak_vehicles   INTEGER,
    avg_congestion  FLOAT,
    needs_police    BOOLEAN DEFAULT FALSE,
    created_at      TIMESTAMP DEFAULT NOW()
);
```

## ⏱️ Event Time vs Processing Time

A key concept in this pipeline is how time is handled for windowing:

| Aspect | Event Time | Processing Time |
|---|---|---|
| Definition | Timestamp embedded in sensor JSON | When Spark receives the message |
| Used in | Q3 — Congestion window | Q2 — Critical alerts |
| Late data | Handled by 10-min watermark | Not applicable |
| Accuracy | Temporally correct windows | Immediate response |
| Priority | Analytical correctness | Operational speed |

**Implementation:**
```python
# Event Time windowing with watermark (Q3)
parsed_df
  .withWatermark("event_time", "10 minutes")
  .groupBy("sensor_id", F.window("event_time", "5 minutes"))
  .agg(F.sum("vehicle_count"), F.avg("avg_speed"))
```

## ⚖️ Ethics & Data Governance

This system handles urban surveillance data. The following governance principles apply:

| Principle | Implementation |
|---|---|
| **Data Minimisation** | Collect only vehicle counts and speeds — no licence plates or personal identifiers |
| **Purpose Limitation** | Data used solely for traffic management and police deployment planning |
| **Retention Policy** | `traffic_events` purged after 90 days; aggregates retained 2 years |
| **Access Control** | Role-based PostgreSQL permissions; audit logging on all queries |
| **Transparency** | Public disclosure of data collected, retention periods, and decisions made |
| **Algorithmic Accountability** | Congestion Index formula and police threshold publicly documented and audited for bias |
